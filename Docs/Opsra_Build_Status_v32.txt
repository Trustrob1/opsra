PATTERN 57 — Worker module-level imports required for patch() to work in tests
All dependencies (get_supabase, service functions) must be imported at MODULE LEVEL.
Lazy imports inside the task function body cannot be patched.

PATTERN 58 — Integration test _ORG_PAYLOAD: permissions nested inside roles
require_permission() reads user["roles"]["permissions"][key] not user["permissions"][key].

PATTERN 59 — _mock_db() chain collision: never set two competing execute.return_value
In _mock_db(), all chained methods return the same chain object. Two mock setups sharing
a path prefix collapse to the same chain.execute.return_value — last write wins.
Rules:
  1. Never set two competing execute.return_value mocks on the same db instance.
  2. Use patch.object(service, "_lead_or_404", return_value=lead) for 404 guards.
  3. Rely on default data=[] where falsy triggers service fallbacks like {**lead, **updates}.
  4. For multiple distinct DB returns, use table.side_effect (see GAP-3 worker tests).
  5. When a new DB call is prepended to a service function, prepend one extra
     MagicMock at position 0 of existing side_effect lists in tests for that function.

PATTERN 60 — Pre-existing tests must be updated when new DB calls are prepended to a function
When a new DB call is added at the START of an existing service function, ALL tests for
that function using side_effect lists will break — sequence shifted by number of new calls.
Fix: prepend one MagicMock per new call at position 0 of the side_effect list.
Update the call order comment. Also: pre-existing worker fixture data must include all
new required fields (e.g. nurture_sequence added to org fixtures after GAP-3).

PATTERN 61 — get_current_org returns "id" not "user_id" for the authenticated user
The dependency returns the raw users table row. User UUID is at org["id"], not org["user_id"].
org["user_id"] does not exist and will raise KeyError at runtime.
Always use org["id"] in route handlers for the authenticated user's UUID.
Integration test _ORG_PAYLOAD must have "id": USER_ID (not "user_id": USER_ID).

PATTERN 62 — db must always be injected via Depends(get_supabase) in route signatures
Never call get_supabase() directly inside a route function body.
Direct calls bypass FastAPI dependency injection — test overrides of get_supabase have no
effect on direct calls, causing SupabaseException: Invalid API key in tests.
Correct:
  async def my_route(org=Depends(get_current_org), db=Depends(get_supabase)):
      result = some_service(db, ...)
Wrong:
  async def my_route(org=Depends(get_current_org)):
      db = get_supabase()   # ← bypasses injection, breaks tests
      result = some_service(db, ...)
Also: SSE generator closures should capture the injected db, not create a second get_supabase() call.
PATTERN 63 — Test patch paths must be derived from the source import, never assumed
Before writing any patch() call in a test, locate the exact import statement in the source
file and derive the patch path from that — never from the calling module name.

Rules:
  "from app.services.X import Y" inside a function body (lazy import) → patch "app.services.X.Y"
  "from app.services.X import Y" at module level                       → patch "app.services.calling_module.Y"
  "import app.services.X"                                              → patch "app.services.X.Y"

Never patch "<calling_module>.<name>" for a lazily imported name — the name never enters the
calling module's namespace.

Verification step (mandatory): Before writing the patch path, open the source file and search
for the imported name. If the import is inside a function body, the patch target is the source
module. If the import is at the top of the file, the patch target is the module under test.

Corollary — mock chain completeness: When a new DB query is added to an existing code path,
all pre-existing test mocks for that path must include the new table name with an explicit
data=[] result AND all chained methods (.neq, .gt, .gte etc) must be configured on the mock
to return the same chain object. A missing chain method returns a fresh truthy MagicMock,
which silently corrupts query results and causes hard-to-diagnose test failures.
Example: get_active_session() uses .neq() and .gt() — any mock missing these will return a
truthy "session" object instead of None, causing early-return branches to fire incorrectly.

## WH-0 FULL SPEC — Intent-First Triage for Unknown WhatsApp Contacts

### Overview

Replace the unconditional "unknown number → create lead → fire qualification bot"
behaviour in webhooks.py with an intent-first triage system. Every first-contact
message from an unrecognised number receives an org-configured WhatsApp Interactive
List Message. The person selects their intent. Only after an explicit selection does
any pipeline action fire. Qualification bot activation requires a deliberate
"I'm interested" selection — it never fires by assumption.

This phase also introduces customer_contacts (B2B employee phone linking),
whatsapp_sessions (triage state machine), contact_type on leads, and
whatsapp_triage_config on organisations.

### Core problem fixed

The system previously had a binary model: known number = route to record,
unknown number = create sales lead. This is wrong for any org whose WhatsApp
number is public-facing. People who can message include: vendors, B2B customer
employees, partners, wrong numbers, and genuine sales prospects. Treating all of
them as sales leads damages reputation and pollutes the pipeline.

### Schema changes

Verify all tables/columns exist before generating SQL (Rule 9).

organisations — two new columns:
  ALTER TABLE public.organisations
    ADD COLUMN whatsapp_triage_config JSONB,
    ADD COLUMN unknown_contact_behavior varchar DEFAULT 'triage_first';
  unknown_contact_behavior valid values: 'triage_first' (default) | 'qualify_immediately'

leads — one new column:
  ALTER TABLE public.leads
    ADD COLUMN contact_type varchar DEFAULT 'sales_lead';
  Valid values: 'sales_lead' | 'business_inquiry' | 'support_contact' | 'other'
  Only 'sales_lead' leads ever enter the qualification pipeline.

handle_session_message(db, org_id, phone_number, session, msg_type,
                       content, interactive_payload, contact_name,
                       now_ts) -> None
  Dispatch by session["session_state"]:

  'triage_sent':
    If msg_type == 'interactive' and interactive_payload has list_reply/button_reply:
      item_id = interactive_payload["list_reply" or "button_reply"]["id"]
      → dispatch_triage_selection(...)
    Else (free text — person typed instead of tapping):
      Re-send the triage menu. Do NOT create a duplicate session.
      send_triage_menu(db, org_id, phone_number, section="unknown")

  'awaiting_identifier':
    If msg_type == 'text' and content:
      → handle_awaiting_identifier(...)

  'active' or anything else: log and return.

dispatch_triage_selection(db, org_id, phone_number, item_id, session,
                          contact_name, now_ts) -> None
  1. Fetch org triage config. Find item matching item_id.
  2. Item not found → treat as free_form.
  3. Dispatch by action:

  'qualify':
    Create lead: contact_type='sales_lead', source=whatsapp_inbound.
    update_session → 'active', selected_action='qualify'.
    Trigger qualification session init (same path as M01-3).

  'identify_customer':
    Send message: "Please share your account email address or company name
    so we can find your record."
    update_session → 'awaiting_identifier', selected_action='identify_customer'.

  'route_to_role':
    Create lead: contact_type from item config (default 'business_inquiry').
    Notify all users in org with roles.template == item["role"] (default 'owner').
    update_session → 'active', selected_action='route_to_role'.
    No qualification bot fires.

  'free_form':
    Create lead: contact_type from item config (default 'other').
    Notify assigned rep or first owner found.
    update_session → 'active', selected_action='free_form'.
    No qualification bot fires.

handle_awaiting_identifier(db, org_id, phone_number, identifier_text,
                           session, contact_name, now_ts) -> None
  1. Python-side search in customers table (Pattern 33):
     Fetch all org customers. Match identifier_text case-insensitive
     against full_name and email.
  2. Found:
     Insert customer_contacts: status='pending'.
     Send: "Thanks — we found your account. A team member will confirm
     your details shortly."
     Notify owners/ops_managers: "New contact pending approval for
     customer [name]."
     update_session → 'active'.
  3. Not found:
     Create lead: contact_type='support_contact', source=whatsapp_inbound.
     Send: "We couldn't find that account. A team member will follow up
     with you shortly."
     Notify owners/ops_managers.
     update_session → 'active'.

## WH-1 FULL SPEC — Customer Intent Classifier

### Overview

WH-0 fixed the unknown-contact pipeline. WH-1 fixes the known-customer pipeline.
Currently, when a known customer sends a WhatsApp message, it is stored and the
assigned rep is notified — nothing else happens automatically. WH-1 adds an intent
classifier that detects what the customer is trying to do and routes accordingly.

This phase also adds context-aware reply detection: the system now knows whether
an inbound message is a reply to a specific outbound trigger (NPS survey, renewal
reminder, drip sequence message) and handles it appropriately instead of treating
all inbound messages as generic free text.

### Core problem fixed

### New function: handle_customer_inbound()

Location: webhooks.py (or customer_service.py — confirm with file before writing).
Called from _handle_inbound_message when customer_id is resolved (not lead_id).

Logic:
  1. Check if message is a reply to a specific outbound context:
     a. NPS survey reply — detect numeric 0-10 or written score in content.
        Write nps_score to customers table. Notify assigned rep/owner.
     b. Renewal reminder reply — detect cancellation or confirmation intent (Haiku call).
        Cancellation intent: create churn alert notification, flag subscription at risk.
        Confirmation intent: notify rep.
     c. Drip sequence reply — any reply to a drip message: pause drip for this customer,
        notify assigned rep. (Drip pause not yet built — store intent, notify only.)
  2. If no outbound context matched, run intent classifier (Haiku call):
     - ticket: auto-create support ticket (contact_type='support_contact' on lead if applicable)
     - billing: notify finance role
     - renewal: notify ops_manager/owner
     - general: store message, notify assigned rep (existing behaviour)
  3. All paths store the inbound message (existing wa_messages logic preserved).
  4. All Haiku calls: S6 (_sanitise_for_prompt), S7 (XML delimiters), S8 (security rules).
  5. Entire function: S14 — no exception may propagate.

### Context detection

Outbound context is determined by checking the last outbound message sent to this
customer (query whatsapp_outbox or wa_messages for most recent outbound within
configurable window, e.g. 48 hours). Message type field (message_type column on
whatsapp_outbox) identifies the trigger:
  'nps_survey' → NPS reply handling
  'renewal_reminder' → renewal reply handling
  'nurture' | 'drip' → drip reply handling
  anything else → intent classifier

### NPS reply parser

detect_nps_score(content: str) -> int | None
  Checks content for a numeric 0-10 reply (exact integer match, stripped).
  Also accepts common written forms: "ten", "zero", etc.
  Returns int 0-10 or None.
  No AI call — pure Python.

On NPS score detected:
  UPDATE customers SET nps_score = <score>, nps_updated_at = now() WHERE id = customer_id.
  Notify assigned rep: "Customer [name] gave an NPS score of [N]."

### Renewal reminder reply classifier

classify_renewal_reply(content: str) -> str  ('cancel' | 'confirm' | 'other')
  Haiku call with short prompt. Returns one of three values.
  S14 — returns 'other' on any failure.

On 'cancel':
  Find active subscription for customer. SET status = 'at_risk' (or equivalent column).
  Notify owners/ops_managers: "Customer [name] indicated cancellation intent in reply
  to renewal reminder."

On 'confirm':
  Notify assigned rep: "Customer [name] confirmed renewal intent."

### Intent classifier

classify_customer_intent(content: str) -> str  ('ticket' | 'billing' | 'renewal' | 'general')
  Haiku call. Returns one of four values. S14 — returns 'general' on failure.

On 'ticket':
  Auto-create ticket: title = first 80 chars of content, status = 'open',
  source = 'whatsapp', customer_id = customer_id.
  Notify assigned rep.

On 'billing':
  Notify all finance role users in org.

On 'renewal':
  Notify ops_manager/owner.

On 'general':
  Existing behaviour: store message, notify rep.

### Mid-pipeline lead stage signals (GAP-C7)

In _handle_inbound_message, for leads in stages: contacted | demo_done | proposal_sent
(not lost, not nurture), run a stage-signal classifier on the inbound message:

classify_lead_stage_signal(content: str, stage: str) -> str
  ('buying' | 'stalling' | 'objection' | 'neutral')
  Haiku call. S14 — returns 'neutral' on failure.

On 'buying':
  Notify assigned rep: "Buying signal detected from lead [name] at [stage] stage."
On 'stalling' | 'objection':
  Notify assigned rep with signal type.
On 'neutral':
  No action beyond existing message store + notify.

## M01-10b FULL SPEC — Aria AI Assistant

### Overview
A persistent, role-aware AI assistant embedded in Opsra as a floating panel
(bottom-right FAB button → slide-in panel). Named "Aria". Delivers morning
briefings, answers ad-hoc questions using real-time role-scoped data, and
provides natural language notification digests. Powered by Claude Haiku.

### Architecture — Three Modes

  Mode 1 — Morning Briefing (pre-generated nightly)
    daily_briefing_worker.py runs at 6am WAT daily.
    For each active user (active in last 30 days):
      Gather role-scoped data snapshot.
      One Haiku call → store in users.briefing_content + users.briefing_generated_at.
    On first login of the day: GET /api/v1/briefing returns { show: true, content }.
    Aria panel opens automatically. Briefing shown as first message.
    User clicks "Let's go" or "Skip" → POST /api/v1/briefing/seen.

  Mode 2 — Assistant Chat (on-demand, Haiku)
    User types question in Aria panel input.
    Backend fetches: last 20 messages from assistant_messages + fresh role-scoped snapshot.
    One Haiku call → stored in assistant_messages.
    Rolling 20-message window. Conversation persists across sessions.
    Messages older than 30 days purged nightly.

  Mode 3 — Smart Notification Digest (twice daily, Haiku)
    Worker runs at 12:00 and 17:00 WAT daily.
    Bundles all unread notifications since last digest into one Haiku call.
    Natural language summary injected as an Aria message.
    FAB button shows badge when digest is ready.
    Existing NotificationsDrawer unchanged.

### Data Scoping Per Role
  Owner / Ops Manager:  pipeline summary, team tasks, ticket counts, subscription health,
                        commission totals, SLA status, team performance
  Sales Agent:          own leads by stage, own tasks due, own demos upcoming,
                        own unread messages
  Customer Success:     assigned customers, churn risks, open tickets, pending tasks
  Support Agent:        assigned tickets, SLA status, open messages
  Finance:              renewals due, payments confirmed, commissions pending
  Affiliate Partner:    own commissions, referred leads status, own tasks

### Cost Model
  Morning briefing:     1 Haiku call/active user/day       ~$0.00044/user/day
  Assistant chat:       ~10 queries/user/day avg            ~$0.01/user/day
  Notification digest:  2 Haiku calls/user/day              ~$0.00088/user/day
  Total/user/day:       ~$0.012   |   100 users: ~$36/mo   |   500 users: ~$180/mo
